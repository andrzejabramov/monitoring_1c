## 1. ВВЕДЕНИЕ

### 1.1. Наименование программы

**Полное наименование:** Система мониторинга нагрузки серверов 1С с функцией биллинга и контроля сессий  
**Краткое наименование:** 1C-Monitor-Stack  
**Версия:** 1.0

### 1.2. Краткая характеристика области применения

Система представляет собой комплекс программных компонентов для мониторинга, анализа и контроля работы серверов 1С:Предприятие и серверов баз данных PostgreSQL.

Система предназначена для автоматизации следующих задач:

- Сбор и хранение метрик нагрузки серверов (CPU, RAM, Swap)
- Мониторинг активных сессий 1С (пользователь, база, клиент, длительность)
- Ведение истории сессий для расчёта стоимости использования ресурсов (биллинг)
- Ограничение количества одновременных сессий на одного пользователя
- Визуализация метрик через дашборды Grafana

**Область применения:**

- Серверы 1С:Предприятие под управлением ОС Linux (Ubuntu)
- Серверы PostgreSQL для хранения данных 1С
- Инфраструктура мониторинга на базе Prometheus + Grafana

**Целевые пользователи:**

- Системные администраторы
- Администраторы баз данных 1С
- Руководители IT-подразделений (для биллинга)
- Пользователи 1С (косвенно, через ограничение сессий)

## 3. НАЗНАЧЕНИЕ РАЗРАБОТКИ

### 3.1. Функциональное назначение

Система состоит из следующих компонентов:

| Компонент                   | Функциональное назначение                                                    |
| --------------------------- | ---------------------------------------------------------------------------- |
| **1C Extension**            | Расширение конфигурации 1С для ограничения количества сессий на пользователя |
| **Python Exporter**         | Скрипт сбора метрик через утилиту `rac`, запись в Prometheus и PostgreSQL    |
| **PostgreSQL (Monitoring)** | Отдельная БД для хранения истории сессий и расчёта биллинга                  |
| **Prometheus**              | Сбор и хранение метрик нагрузки серверов (CPU, RAM, Swap, сессии)            |
| **Grafana**                 | Визуализация метрик и дашборды для анализа нагрузки                          |
| **Docker Compose**          | Оркестрация всех компонентов системы (разработка и продакшн)                 |

### 3.2. Эксплуатационное назначение

Система является компонентом инфраструктуры мониторинга серверов 1С:Предприятие, позволяющей:

- Контролировать нагрузку на серверы 1С и PostgreSQL в реальном времени
- Вести историю сессий для последующего анализа и биллинга
- Ограничивать злоупотребление ресурсами (множественные сессии одного пользователя)
- Снижать риски перегрузки серверов в пиковые часы
- Предоставлять прозрачную отчётность по использованию ресурсов

Система эксплуатируется в составе инфраструктуры мониторинга и администрирования серверов 1С.

## 4. ТРЕБОВАНИЯ К ПРОГРАММЕ

### 4.1. Требования к функциональным характеристикам

#### 4.1.1. Общая архитектура

Система состоит из следующих компонентов:

| Компонент            | Назначение                                        | Язык / Технология   |
| -------------------- | ------------------------------------------------- | ------------------- |
| **1C Extension**     | Контроль сессий на уровне платформы 1С            | 1C:Enterprise (BSL) |
| **onec_exporter.py** | Сбор метрик через `rac`, запись в БД и Prometheus | Python 3.8+         |
| **monitoring-db**    | Хранение истории сессий для биллинга              | PostgreSQL 15       |
| **Prometheus**       | Хранение метрик нагрузки (временные ряды)         | Prometheus 2.x      |
| **Grafana**          | Визуализация дашбордов                            | Grafana 10.x        |
| **Docker Compose**   | Оркестрация контейнеров                           | Docker Compose 3.8  |

#### 4.1.2. Требования к компоненту 1C Extension (ограничение сессий)

| №   | Требование                                                                                      |
| --- | ----------------------------------------------------------------------------------------------- |
| 1   | Расширение должно перехватывать событие `ОбработчикНачалаСеанса`                                |
| 2   | Расширение должно проверять количество активных сессий текущего пользователя                    |
| 3   | Расширение должно блокировать вход при превышении лимита (по умолчанию: 1 сессия)               |
| 4   | Расширение должно исключать системных пользователей из проверки (Admin, srv_1c, и др.)          |
| 5   | Расширение должно записывать попытки нарушения в Журнал Регистрации 1С                          |
| 6   | Расширение должно выводиться с понятным сообщением об ошибке                                    |
| 7   | Расширение должно применяться без модификации основной конфигурации (через механизм Расширений) |

**Синтаксис события:**

```bsl
Процедура ОбработчикНачалаСеанса() Экспорт
    // Проверка количества сессий
    // Блокировка при превышении лимита
КонецПроцедуры
```

#### 4.1.3. Требования к компоненту Python Exporter (сбор метрик)

| №   | Требование                                                                                              |
| --- | ------------------------------------------------------------------------------------------------------- |
| 1   | Скрипт должен опрашивать сервер 1С через утилиту `rac session list --format=json`                       |
| 2   | Скрипт должен отдавать метрики в формате Prometheus на порту 9111                                       |
| 3   | Скрипт должен записывать события начала/окончания сессий в PostgreSQL (таблица `onec_session_log`)      |
| 4   | Скрипт должен работать в цикле с интервалом опроса 60 секунд                                            |
| 5   | Скрипт должен определять новые и завершённые сессии путём сравнения с предыдущим состоянием             |
| 6   | Скрипт должен записывать метрики: ИБ, Пользователь, Клиент, Время начала, Время окончания, Длительность |
| 7   | Скрипт должен обрабатывать ошибки подключения к `rac` без прерывания работы                             |
| 8   | Скрипт должен запускаться как systemd-сервис в продакшн-среде                                           |

**Метрики Prometheus:**

```
onec_active_sessions{infobase, user, client, license}
onec_session_duration_seconds{infobase, user, session_id}
onec_licenses_used{type}
```

#### 4.1.4. Требования к компоненту PostgreSQL (Monitoring DB)

| №   | Требование                                                                          |
| --- | ----------------------------------------------------------------------------------- |
| 1   | БД должна быть развёрнута в отдельном контейнере (не в БД 1С!)                      |
| 2   | БД должна содержать таблицу `onec_session_log` для истории сессий                   |
| 3   | БД должна содержать таблицу `onec_user_limits` для настройки лимитов пользователей  |
| 4   | БД должна иметь индексы для быстрых выборок по пользователю и времени               |
| 5   | БД должна быть доступна на порту 5433 (чтобы не конфликтовать с другими PostgreSQL) |
| 6   | БД должна иметь резервное копирование (через Docker volumes)                        |

**Схема таблицы `onec_session_log`:**

```sql
CREATE TABLE onec_session_log (
    id SERIAL PRIMARY KEY,
    session_id VARCHAR(50) NOT NULL,
    user_name VARCHAR(100) NOT NULL,
    infobase_name VARCHAR(100),
    client_type VARCHAR(50),
    start_time TIMESTAMP NOT NULL,
    end_time TIMESTAMP,
    duration_seconds INTEGER,
    avg_server_cpu_percent FLOAT,
    created_at TIMESTAMP DEFAULT NOW()
);
```

| №   | Требование                                                            |

### 4.2. Требования к надежности

#### 4.2.1. Требования к обеспечению надежного функционирования

| №   | Требование                                                                  |
| --- | --------------------------------------------------------------------------- |
| 1   | Система должна продолжать работу при временной недоступности 1С-сервера     |
| 2   | Python Exporter должен логировать все ошибки с кодами возврата              |
| 3   | Расширение 1С не должно блокировать вход при ошибке получения списка сессий |
| 4   | Все компоненты должны иметь restart policy `unless-stopped`                 |
| 5   | Данные PostgreSQL должны сохраняться при перезапуске контейнеров            |

#### 4.2.2. Время восстановления после отказа

| Сценарий             | Требование                                                                      |
| -------------------- | ------------------------------------------------------------------------------- |
| Сбой Python Exporter | Автоматический перезапуск через systemd/Docker, потеря не более 1 минуты метрик |
| Сбой расширения 1С   | Вход пользователей не блокируется, логируется ошибка                            |
| Сбой PostgreSQL      | Данные сохраняются в volume, восстановление при старте контейнера               |
| Сбой Prometheus      | Потеря метрик за период недоступности, исторические данные сохраняются          |

#### 4.2.3. Отказы из-за некорректных действий оператора

| №   | Требование                                                                   |
| --- | ---------------------------------------------------------------------------- |
| 1   | Удаление контейнеров не должно приводить к потере данных (volumes)           |
| 2   | Изменение конфигурации должно требовать перезапуска соответствующего сервиса |
| 3   | Пароли должны храниться в `.env` файле, не в коде                            |
